## WS6 Frontend Boundary Pilot Note

This pilot suggests future underweighted-subsystem refinements should prefer evidence-gated class rules over path-name specials.

- Effective pattern here: boost a frontend or product-web boundary only when it is both substantial and runtime-like, using cheap structural signals such as authored source volume, a colocated Node package manifest, and frontend framework config or route roots.
- Why it held up on Dify: the rule let `web/` surface as a first-class orientation boundary while keeping the primary backend entrypoints ahead of it.
- Guardrail to preserve: do not boost thin helper web clients, SDK wrappers, or docs sites on naming alone; require multiple reinforcing signals before changing rank.
